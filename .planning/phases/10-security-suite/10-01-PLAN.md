- `load_block_cache(&self, entries: Vec<BlockEntry>)` — write-lock block_cache + replace

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SIP message → SecurityState | Untrusted IP/realm from incoming SIP message enters flood + brute-force counters |
| REST API → firewall cache | Operator-supplied CIDR rules enter the in-memory firewall rule cache |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01-01 | Denial of Service | SecurityState DashMap (flood tracker) | mitigate | Flush task evicts stale WindowState entries (window expired > 2x duration) to bound memory growth under spoofed-IP DDoS (RISK-05) |
| T-10-01-02 | Tampering | firewall_rules RwLock cache | mitigate | Update cache ONLY after DB transaction commits in PATCH handler (Wave 2); hot path reads cache only — never a stale-empty window (RISK-06) |
| T-10-01-03 | Information Disclosure | security_state.rs FloodEntry/AuthFailureEntry | accept | These are operational stats read by authenticated operators only; no PII beyond IP address which is already in CDRs |
| T-10-01-04 | Spoofing | IP extraction from Via header | mitigate | Use exact acl.rs Via extraction pattern (RISK-01): `SipConnection::parse_target_from_via` — do not use connection-level remote_addr which can be spoofed in UDP |
</threat_model>
